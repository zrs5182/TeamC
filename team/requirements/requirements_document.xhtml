Argument Mapping Software Requirements

Team C
Created: 18 March 2013

Status of This Document

This draft is under review by Truman State University: CS 370: Team C, for use by Truman State University for the purpose of increasing the critical thinking abilities of students through the use of argument mapping. This document is subject to change at any time, and may be updated, replaced, or obsoleted at any time.

This document defines the high-level requirements for the Argument Mapping Software being designed for Truman State University. The requirements are categorized as follows:

  • 1. General Requirements
  • 2. Appearance
  • 3. User Interface
  • 4. Utility
  • 5. Optional

1. General Requirements

(a) A new file contains a single claim box (1.7)
(b) The system must be able to add support and contradiction premises to any claim from the canvas (3.1, 3.7)
(c) Child claims and their sub-arguments must be able to change parents (1.7)
(d) Claims must be removable (1.7)
(e) When claims are deleted, their children are also deleted (1.7)
(f) The system must be able to add co-premises (1.7)
(g) The system must be able to join premises together to make co-premises (1.7)
(h) The system must be able to separate co-premises into individual premises (1.7)
(i) The system must be able to change the type of a claim (3.7)
(j) Refutations of refutations must be automatically displayed as rebuttals (1.7)
(k) All claims other than the original argument must connect to parent claims (3.6)

2. Appearance

(a) The system must label claims (1.3, 3.1, 3.9)
(b) Connections must be curvy(1.3, 3.5)
(c) All corners must be rounded (3.5)
(d) Claims must not overlap visually (1.3)
(e) The system must be able to visually identify co-premises (1.3)
(f) The system must have different connections for support and refutation (1.3)
(g) The rebuttal type must be identified by a unique color (1.3)
(h) Spatial arrangement of claims must be handled by the software automatically (3.1)

3. User Interface

(a) The system must support mouse control for moving claims to a new parent (3.4)
(b) The system must support mouse control for view control (3.4)
(c) The system must support mouse control for administrative tasks (3.4)
(d) The system must support keyboard control for text entry for claims (3.4)
(e) The diagram must persist even when outside the viewable area (3.4, 3.7)
(f) The view must be scalable and pannable (3.4)

4. Utility

(a) The system must be cross platform (1.5, 1.7, 2.1)
(b) The system must be able to complete the Rationale tutorials using the system (1.7)
(c) State of the diagram must be able to be saved in a persistent, sharable format (1.6, 1.7, 3.1, 3.3, 3.7)
(d) Saved maps must be able to be re-opened and edited (1.6, 3.3)
(e) The system must be able to export the map to a scalable, printable format (1.7, 2.2, 3.3)
(f) The system must run on the Truman network infrastructure (1.2, 2.1, 2.5)
(g) The system must provide color customization options to accomodate colorblind users. (1.7, 3.1)

5. Optional

(a) The system must be able to support multiple users at a single time. (3.2)
(b) The system must have a support manual that explains how to use each component of the system. (3.10)
(c) The system must output the file in an ASCII file format. (3.3)
(d) The system must support the ability to annotate a project file. (1.8, 3.3)
(e) The system must be able to run on mobile devices. (2.1.a)
