1) What is the most important function of the program?

  To provide, a SIMPLE, intuitive, smooth interface for creating aesthetically polished argument maps.

2) Who are the primary users of the program, and what is their level of technical knowledge?

  Students and professors.

3) Can you draw an example argument map the way you would want the program to produce it?

  Diagram similar to Rationale tutorial diagrams (boxes, connectors, appearances)

 A) What elements of an argument do you wish to be represented? (e.g., thesis, premises, objections, etc)

4) How much, if any, of this project is already underway?

  A student team has tested argument mapping with various available software.

5) Have you considered repurposing an already available product (or products) to meet your needs?

  Rationale is expensive and not sufficiently cross-platform capable. Using other graphics software is possible, but the client desires a dedicated application.

6) Should we provide templates?

  Sure, provide 'this is what an argument is' type templates.

7) What are the must-have features?

  Cross-platform, makes aesthetic diagrams with little effort, features to work through the tutorials on the Rationale website, saving the diagram, ability to print/make pdf, able to submit electronically or via hard copy, color settings.

8) Should there be extra meta data attachable to the map?

  Sticky-note type comments for collaboration/grading/tutoring would be great.

 

1) Would you prefer a standalone desktop application or a web application (in-browser, like Google Docs)?

  Either is fine as long as the product is cross-platform capable. Prefer to stay away from volatile or soon-to-be obsolete platforms like Java or Flash. OS-independent environment like JS or HTML5 preferred.

 A) If desktop, what platform(s) do you require it to run on? (Windows, Mac OS, Linux, mobile)

    Windows, Mac, and Linux compatibility are required. Mobile/touchscreen compatibility considered a major plus for potential longevity.

2) What file format(s) should the file produce?

  At least one pdf or image format for visual presentation/printing.

3) Are there any security concerns we should be aware of?

  Files created by the program are not subject to security concerns. Authentication through/hosting on Truman network may eventually be an issue if the product is a web app.

4) What do you deem an acceptable trade-off between schedule and quality?

 Development time is basically confined to this semester due to class schedule.

5) Who will be using the software?

  Students will be the main users, but anyone should be able to use it.

6) Is anything likely to change?

  Increasing tablet use, might need more complex argument types or new types of boxes/connectors.

 

1) Pretend you are using the program, finished as you envision it. Walk step-by-step through performing some basic functions, sketching rough diagrams where necessary.

  Create boxes with text and label, connect boxes with appropriate arrows. Save map and manipulate file format as desired, share (possibly via email). Funcionalities are sparse but sufficient; lightweight. Snap-to-grid. Customizable label colors.

2) Is the program intended for a single user at a time?

  Real-time collaboration would be appreciated, but is not a requirement due to complexity.

3) Are there any special operations you want to be able to perform on the data? (e.g., exporting to a presentation program like PowerPoint)

   Some form of persistent saves MUST be available. Not required, but pluses: working project filetype, text output for accessibility (XML), saving maps as templates, annotations for grading. Analysis functionality is NOT desired.

4) What will the user's primary method of interaction with the program be?

 Mouse and keyboard OR touchscreen. Emphasis on canvas area, as in image editing software. Scroll wheel to zoom, click and drag to pan. Elements/color selection in palette toolbar/window. Design with touchscreen accessibility in mind.

5) What do you like about Rationale?

  The push of an argument from the prettiness of the way lines connect.

6) What do you dislike about Rationale?

  It is hard to hook things together. It includes too much, more than what we need or want.

7) What are the components of the interface?

  A canvas to work on (mouse grab/zoom), support/opposition boxes (right click to change properties), button to save, button to share.

8) Should we guide the user?

  No, let them make mistakes.

9) Should the premises be labeled?

  Yes.

10) Should there be support documentation?

Tooltips for the interface tools, but it's up to the teachers to figure out how to make use of the software.
